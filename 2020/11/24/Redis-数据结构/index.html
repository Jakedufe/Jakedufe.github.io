Redis 数据结构
更新于 2020-11-24 Redis 热度 1 喜欢 0

摘要

总结Redis中使用到的存储数据结构。

SDS

Redis没有直接使用C语言的字符串，而是引入了简单动态字符串（simple dynamic string，SDS）作为Redis的默认字符串表示。

在Redis中，C字符串只会作为字符串字面量用在一些值不会进行修改的场景，比如打印日志。

SDS除了用来保存数据库中的字符串值之外，还被用作缓冲区：

  • AOF模块中的AOF缓冲区
  • 客户端状态中的输入缓冲区。

SDS定义

每个sds.h/sdshdr结构表示一个SDS值：

SDS与C字符串的比较（优点）：

  1. 常数复杂度获取字符串长度

  2. 杜绝缓冲区溢出（先判断空间，再执行命令）

  3. 减少修改字符串时带来的内存重分配次数

    空间预分配：如果对SDS进行修改之后，SDS的长度将小于1MB，那么程序分配和len属性同样大小的未使用空间。如果对SDS进行修改之后，SDS的长度将小于1MB，那么程序分配1MB的未使用空间。

    惰性空间分配，也就是要存储的字符串数量小于free属性时，并不会触发重分配。

  4. 二进制安全

    因为C字符串通过空字符来标识字符串的末尾，所以并不能存储任意类型的数据。但是SDS标注了存储长度len，所以是二进制安全的，

  5. 兼容部分C字符串函数

Dict

字典是Redis最核心的数据结构，在实现中被大量使用。

字典的实现

Redis字典所使用的数据结构定义在dict.h中。

下文提供了四个structure的源码，dict为Redis的字典，其中ht数组指向两个dictht，dictht是hashtable的数据结构，dictht中的table又包括一个dictEntry数组，数组中每一个位置散列成一个dictEntry链表。

dictEntry的v属性是键值对中的值，v是C语言中的union，可以是一个指针、或者是一个uint64_t整数，又或者是一个int64_t整数。

dict中的type和privdata搭配起来使用，是针对不同类型的键值对，为创建多态字典而设置。type保存的是针对字典数据结构的一系列操作，和Linked List结构的dup、free、match属性作用其实是一样的，只是字典的type比较多，所以专门用一个structure来表示，privdata保存了需要传给那些类型特定函数的可选参数。dict中之所以存在ht数组和rehashidx，是为了在hash表需要扩容时配合使用。rehashidx相当于扩容时的计数器，而hb[1]则是扩容过程中的新表。

Rehash

步骤：

  1. 为ht[2]分配空间，如果要执行的是扩展操作，则ht[1]的大小为第一个大于ht[0].used*2的$2^{n}$。如果是收缩操作，ht[1]的大小为第一个大于ht[0].used的$2^{n}$。
  2. 将保存在ht[0]的键值对rehash到ht[1]上面。
  3. 当所有键值对迁移后，释放ht[0]，将ht[1]设置为ht[0]，创建空的ht[1]为下一次rehash做准备。

渐进式rehash

第二步中的复制操作，并不是一次性、集中式完成的，而是分多次、渐进式地完成的。这是为了保证服务器对外能够持续提供服务。

渐进式rehash的详细步骤：

  1. 为ht[1]分配空间。
  2. 在字典中维持一个索引计数器rehashidx，并将其值设为0，表示rehash工作正式开始。
  3. 在rehash期间，每次对字典中的键执行操作，会顺便将键值对rehash到ht[1]，并且rehashidx的值增一。
  4. 当所有键值对迁移后，释放ht[0]，将ht[1]设置为ht[0]，创建空的ht[1]为下一次rehash做准备，将rehashidx值设为-1.

渐进式rehash执行期间的哈希表操作

先查ht[0],没有的话再去查[1]。

hashtable的扩展与收缩

当以下条件的任意一个被满足时，程序会自动开始对哈希表执行扩展操作：

  1. 服务器目前没有在执行BGSAVE或者BGREWRITEAOF命令，并且负载因子大于等于1。
  2. 服务器目前正在执行BGSAVE或者BGREWRITEAOF命令，并且负载因子大于等于5。

负载因子之所以不同，是因为执行BGSAVE或者BGREWRITEAOF命令，Redis需要创建当前服务器子进程来执行，大多数系统采用写时复制来优化子进程使用效率，rehash导致父子进程不同，会导致子进程需要更多的内存。

当哈希表的负载因子小于0.1时，程序自动开始对哈希表执行收缩操作。

Skip List

跳跃表（skiplist）是一种有序的数据结构，它通过在每个节点中维护多个指向其他节点的指针，从而达到快速访问节点的目的。跳跃表是我之前没有学习过的一种数据结构，这里只给出它的一些特点，我将在另一篇专门的文章中通过阅读论文《Skip Lists: A Probabilistic Alternative to Balanced Trees》来理解它的基本定义和基础算法。

跳跃表支持平均O（logN）、最坏O（N）复杂度的节点查找，还可以通过顺序性操作来批量处理节点。跳跃表的效率可以和平衡树相媲美，并且实现更为简单。

Redis只在两个地方用到了跳跃表，一个是实现有序集合键，另一个是在集群中用作内部数据结构。

跳跃表的实现

跳跃表是由redis.h/zskiplistNode和redis.h/zsiplist两个数据结构实现的。

多个跳跃表节点组成一个跳跃表，由跳跃表结构体管理一个跳跃表，并保存一些重要信息，比如头节点、尾节点、最大层高（表头节点的层高不计算在内）和跳跃表节点数量（表头节点不计算在内）。

整数集合

整数集合（intset）是集合键的底层实现之一，当一个集合只包含整数数值元素，并且这个集合的元素数量不多时，Redis就会使用整数集合作为集合键的底层实现。

这是一种类似与JVM处理128以内的Integer的处理方式。

实现

每个intset.h/intset结构表示一个整数集合：

contents数组是整数集合的底层实现：整数集合的每个元素都是数组的一个数组项，各个项按值从小到大有序排列，并且不包含重复项。编码方式决定数组存储的数组项类型。

升级

在添加新元素到整数集合时，如果新元素的长度比整数集合原来所有元素的长度长，整数集合要先进行升级。

升级要为contents重新分配空间、将原有元素转化为与新元素相同的类型，然后将新元素添加进数组。

升级的好处

  1. 提升整数集合的灵活性。放入整数集合时，不用考虑数据类型。
  2. 节约内存。

不支持降级

压缩列表

压缩列表（ziplist）是列表键和哈希键的底层实现之一。当一个列表键只包含少量列表项，并且每个列表项要么就是小整数值，要么就是长度比较短的字符串，那么Redis就会使用压缩列表来做列表键的底层实现。

压缩列表的构成

压缩列表是Redis为了节约内存而开发的，是由一系列特殊编码的连续内存块组成的顺序型（sequential）数据结构。一个压缩列表可以包含任意多个节点，每个节点可以保存一个字节数组或者一个整数值。

zlbytes记录整个压缩列表占用的内存字节数，zltail记录压缩列表表尾节点距离列表的起始地址有多少字节。zllen记录了压缩列表包含的节点数量，entryX是列表节点，zlend是特殊值0xFF，标记压缩列表的末端。

压缩列表节点的构成

每个压缩列表节点可以保存一个字节数组或者一个整数值，字节数组有三种长度，整数值有六种长度。每个压缩列表节点都由previous_entry_length、encoding、content三个部分组成。

previous_entry_length

节点的previous_entry_length属性以字节为单位，记录了压缩列表中前一个节点的长度。取决于前一个节点的长度是否小于254字节，previous_entry_length属性的长度可以是1字节或者5字节。

previous_entry_length使得压缩列表可以从表尾向表头遍历操作。

encoding

encoding属性记录了节点的content属性所保存数据的类型和长度。即通过encoding可以区分是字节数组三种长度，整数值六种长度中的具体哪一种。

content

节点的content属性负责保存节点的值，值的类型和长度由encoding决定。

连锁更新(不重要)

连锁更新描述的是这样一种情况，previous_entry_length属性会根据前一个节点是否小于254字节而选择使用1字节还是5字节保存前一个节点的长度。现在，如果在一个压缩列表中，有多个连续的、长度介于250字节和253字节之间的节点。因为这些节点的长度小于254字节，所有previous_entry_length使用1字节存储，这时，在这些节点之前加入一个长度大于等于254字节的节点，会导致后一个节点的空间重分配，后一个节点的长度大于等于254字节，继而影响后续的节点，这就是连锁更新。

除了添加新节点可能会引发连锁更新之外，删除节点也会可能引发连锁更新。

因为连锁更新在最坏情况下需要对压缩列表执行N次空间重分配，每次是O（N），所以连锁更新的最坏复杂度为O（$N^{2}$）。
